# 5. 支援追加・変更・停止

## 概要

会員さまがマイページから、事務局を介さずに
**新しい支援の追加／支援内容の変更（馬・口数）／支援の停止**を
すべて完結できる導線を実装いたしました。

操作はいずれも裏側で Stripe と同期される前提で設計しており、
申込内容の確定・変更・停止予定が、決済状態や次回請求にも連動する構成としております。
また、単にフォームを置くだけではなく、
会員さまが途中で迷わないよう、確認ステップや補助文言を追加し、
誤操作を防止する導線にしております。

## 実装した主な機能

- 「新しい支援を追加」: 対象馬・プラン・口数を 3 ステップで選択し、最終確認後に申込
- 「支援を変更」: 変更前・変更後・差額・次回請求への影響を確認してから確定
- 「支援を停止」: 次回更新日で停止を標準とし、停止予定日を会員へ明示
- 同一会員が同じ馬を複数申込した場合は、既存支援へ口数を統合し、重複行を作らない
- すべての操作は `audit_logs` に記録し、管理者側で履歴として追跡可能
- Stripe 同期失敗時は DB 側だけ確定させず、未確定状態のままエラーを返す安全設計

## 実装コード（抜粋）

### 支援の新規追加

```ts
const { data: existingRow } = await admin
  .from("support_subscriptions")
  .select("id, units, monthly_amount, stripe_subscription_item_id, horse:horses(name)")
  .eq("customer_id", session.customerId)
  .eq("horse_id", horse.id)
  .in("status", ["active", "past_due"])
  .maybeSingle();

const { data: inserted } = await admin
  .from("support_subscriptions")
  .insert({
    contract_id: contractId,
    customer_id: session.customerId,
    horse_id: horse.id,
    units,
    monthly_amount: monthly,
    status: "incomplete",
  })
  .select("id")
  .single();
```

### Stripe 同期成功後のみ確定

```ts
const canActivate = sync?.synced || !stripeEnabled;
await admin
  .from("support_subscriptions")
  .update({ status: canActivate ? "active" : "incomplete" })
  .eq("id", inserted.id);

if (!canActivate) {
  return NextResponse.json(
    { error: "決済の同期に失敗したため、支援の登録を確定できませんでした。" },
    { status: 502 },
  );
}
```

### 支援停止（次回更新日で停止）

```ts
sync = await syncSupportCancel({
  stripe_subscription_item_id: existing.stripe_subscription_item_id ?? null,
  stripe_subscription_id: existing.contract?.stripe_subscription_id ?? null,
  immediate: false,
});

const scheduledAt =
  sync?.scheduled_cancel_at ??
  existing.contract?.current_period_end ??
  null;

if (scheduledAt) {
  await admin
    .from("support_subscriptions")
    .update({ canceled_at: scheduledAt })
    .eq("id", params.id);
}
```

## 画面で確認できるポイント

支援追加では、いきなり申込を確定するのではなく、
対象馬の選択、口数の選択、内容確認という順序で進むため、
高齢の会員さまでも現在何を決めているのかが分かりやすい設計となっております。

支援変更では、変更前と変更後の内容を比較し、
月額の変化や差額の説明を確認した上で確定できるため、
「どの程度金額が変わるのか」が不明なまま手続きが進むことを防いでおります。

支援停止については、即時停止ではなく、
原則として次回更新日で終了する形式を採用しております。
これにより、会員さまは「すぐに特典が使えなくなるのではないか」という不安を持たずに、
終了予定日を確認したうえで停止手続きを行えます。

## 実装上の意図

この機能群では、会員さまが自分で手続きを完了できることと、
運営側で後から追跡・復旧できることの両立を重視しました。
そのため、内部的には Stripe 同期前に一旦未確定状態を持ち、
同期成功後にのみ `active` へ昇格させる構成を採用しております。

また、停止についても、会員さまの誤操作リスクを下げるため、
「次回更新日で停止」を標準とし、
まだ利用可能な期間が残っていることを UI 上で明示しています。
この方針により、問い合わせ削減と運営負荷軽減の両方に配慮した実装になっております。

## ご依頼内容との対応

| いただいたご要望 | 対応状況 |
| --- | --- |
| 事務局を介さずに会員が変更・停止できるようにしたい | 全操作をマイページ内で完結できる導線を実装 |
| 口数変更が Stripe にも反映されるようにしたい | Stripe 同期処理を介して次回請求に反映される構成を実装 |
| 停止は次回更新日で分かりやすく止めたい | `cancel_at_period_end` を基準に停止予定として処理 |
| 同じ馬の重複申込を自動で統合したい | 既存支援がある場合は新規行を増やさず口数統合 |
| Stripe 同期に失敗した場合も安全にしたい | 同期失敗時は未確定状態を保持し、確定せずエラー返却 |
