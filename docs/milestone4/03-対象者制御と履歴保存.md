# 3. 対象者制御と履歴保存

## 概要

Milestone 4 では、見学会・個別見学の対象者制御と、予約・イベント操作の履歴保存を実装いたしました。

支援者限定イベントを設定できるようにし、支援中の会員さまだけが対象イベントを予約できる仕組みを整えています。また、予約作成、変更、キャンセル、削除、イベント作成、更新、非公開化、削除といった操作を監査ログとして保存し、管理画面から確認できるようにしました。

## 対象者制御の実装内容

| 項目 | 状態 | 内容 |
| --- | --- | --- |
| 支援者限定設定 | 完了 | イベントごとに `supporters_only` を設定可能 |
| 支援状態判定 | 完了 | `support_subscriptions` に有効な支援があるかを判定 |
| 会員画面での対象外表示 | 完了 | 対象外のイベントは予約ボタンを出さず「対象外」と表示 |
| API 側の予約制御 | 完了 | UI だけでなく API 側でも支援者限定チェックを実施 |
| 管理者上書き | 完了 | 必要時のみ `bypass_guard` で管理者が強制登録可能 |

## 支援者限定イベント

イベントには `supporters_only` という項目を持たせています。

この値が有効なイベントは、現在支援中の会員さまだけが予約できます。支援中かどうかは、`support_subscriptions` に `status = active` のレコードがあるかで判定しています。

会員画面では、支援者限定イベントに「支援者限定」ラベルを表示します。支援中でない会員さまには「対象外」と表示し、予約操作をできないようにしています。

## API 側の制御

対象者制御は画面表示だけではなく、API 側でも実施しています。

会員が予約を作成する `POST /api/mypage/bookings` では、対象イベントが支援者限定の場合、`hasActiveSupport()` によって会員の支援状態を確認します。支援中でない場合は予約を拒否します。

これにより、画面を経由しない直接 API アクセスでも、不正な予約を防止できます。

## 管理者による上書き

管理者さまが電話対応や個別事情により特別に予約を追加する必要がある場合に備え、管理者予約 API では `bypass_guard` に対応しています。

通常は支援者限定チェックと定員チェックを行いますが、管理者が明示的に上書きした場合のみ、例外的に登録できます。この操作も監査ログに記録されるため、後から経緯を確認できます。

## 履歴保存の実装内容

| 項目 | 状態 | 内容 |
| --- | --- | --- |
| 予約作成履歴 | 完了 | 会員・管理者の予約作成を記録 |
| 予約変更履歴 | 完了 | 人数、メモ、状態変更の前後を記録 |
| 予約キャンセル履歴 | 完了 | 会員・管理者のキャンセル操作を記録 |
| 予約削除履歴 | 完了 | 管理者による完全削除を記録 |
| イベント作成履歴 | 完了 | 管理者によるイベント作成を記録 |
| イベント更新履歴 | 完了 | イベント設定の変更前後を記録 |
| イベント非公開化・削除履歴 | 完了 | 公開停止・削除操作を記録 |
| 管理画面での閲覧 | 完了 | 監査ログ画面・イベント詳細画面から確認可能 |

## 監査ログの保存内容

履歴は `audit_logs` テーブルに保存しています。

主な保存項目は以下です。

- 操作者
- アクション名
- 対象テーブル
- 対象 ID
- 操作日時
- 変更前後の内容
- 予約人数
- イベント ID
- メモ
- 管理者上書きの有無

操作ログは `writeAudit()` ヘルパーに集約し、各 API から共通的に呼び出しています。監査ログの保存に失敗した場合でも、予約操作そのものが失敗しないように設計しています。

## 記録される主なアクション

| アクション | 内容 |
| --- | --- |
| `booking.self_create` | 会員による予約作成 |
| `booking.self_update` | 会員による予約変更 |
| `booking.self_cancel` | 会員による予約キャンセル |
| `booking.create` | 管理者による予約作成 |
| `booking.update` | 管理者による予約変更 |
| `booking.cancel` | 管理者による予約キャンセル |
| `booking.delete` | 管理者による予約削除 |
| `event.create` | イベント作成 |
| `event.update` | イベント更新 |
| `event.unpublish` | イベント非公開化 |
| `event.delete` | イベント削除 |

## 管理画面での履歴確認

管理者さまは `/admin/audit-logs` から、予約・イベントを含む監査ログを確認できます。

また、イベント詳細画面 `/admin/events/[id]` では、対象イベントに紐づく以下の履歴を確認できます。

- 予約変更・キャンセル履歴
- イベント設定変更履歴
- 操作日時
- 操作種別
- 対象会員
- 変更内容

これにより、予約トラブルや問い合わせが発生した場合でも、管理画面から操作履歴を追跡できます。

## 完了確認

以下の動作を確認済みです。

- 支援者限定イベントを作成できること
- 支援中会員だけが支援者限定イベントを予約できること
- 支援中でない会員は対象外として扱われること
- API 直接実行でも対象者制御が働くこと
- 予約作成・変更・キャンセルが監査ログに保存されること
- イベント作成・更新・非公開化・削除が監査ログに保存されること
- 管理画面から履歴を閲覧できること

以上により、Milestone 4 における対象者制御と履歴保存は完了しております。
